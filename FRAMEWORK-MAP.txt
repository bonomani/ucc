FRAMEWORK MAP
=============

This document is the entry point for the UCC framework.
Read this first. It shows the complete structure in one page.

------------------------------------------------------------

1. WHAT THIS FRAMEWORK IS
------------------------------------------------------------

A set of orthogonal concepts for building correct, observable,
and auditable software interactions — independent of language,
protocol, or domain.

It does NOT define:
- frameworks or libraries
- execution engines
- development processes

------------------------------------------------------------

2. THE TWO ORTHOGONAL AXES
------------------------------------------------------------

AXIS A — Interaction Type              (what kind of interaction is this?)
----------------------------
  GIC  Observable fact                 no obligation, no guarantee
  GCC  Normative contract              explicit result, external responsibility
  UCC  Convergence contract            idempotent, retry-safe, state-based

  GIC → GCC → UCC is a strict hierarchy. Each level adds invariants.
  Escalation must be explicit. (see BISS.txt, GIC.txt, GCC.txt)

AXIS B — Internal Quality Grade        (how rigorous is the implementation?)
----------------------------
  Basic  Minimal structure             no formal invariants
  RIG    Rigorous Interaction Grade    invariants, observability, tests
                                       — NOT a contract, freely refactorable

  These axes are INDEPENDENT.
  Any interaction type can be implemented at any quality grade.
  (see RIG.txt)

------------------------------------------------------------

3. NOTE ON ORTHOGONALITY
------------------------------------------------------------
The four-tier UCC spec structure (Tier 0–3) and the BISS two-axis model
are orthogonal to each other and must not be conflated:

  BISS Axis A (GIC → GCC → UCC) — describes the TYPE of interaction contract.
  BISS Axis B (Basic / RIG)      — describes the QUALITY GRADE of an implementation.
  UCC Tiers 0–3                 — describes the INTERNAL STRUCTURE of the UCC spec.

The UCC tiers are a documentation architecture, not a runtime hierarchy.
A component that uses UCC-EXT-ASYNC (Tier 2) is still a UCC component (Axis A),
and may be either Basic or RIG grade (Axis B). The axes and the tiers
vary independently.

------------------------------------------------------------

4. RECORD MODEL (CROSS-CUTTING)
------------------------------------------------------------

All interactions produce immutable records.
Records describe facts — they never control execution.

Mandatory fields:  event_id, type, correlation_id, timestamp_utc
Optional:          parent_event_id (causality), payload (typed by record type)

Record types: intent, execution, state-declaration, decision, policy, quality, view
(See GLOSSARY.txt for the distinction between Intent and Declaration.)
(see RECORD-MODEL-SPEC.txt)

------------------------------------------------------------

5. CORE CONCEPTS (VOCABULARY)
------------------------------------------------------------

All documents share these concepts. See GLOSSARY.txt for definitions.

  Event             immutable fact that something happened
  Intent            correlation group (NOT desired state — see GLOSSARY)
  Declaration       desired state specification (UCC term)
  Target            the object an interaction applies to
  State Reference   declared read/write/assume access to system state
  Decision          recorded choice among alternatives
  Reasoning Artifact  versioned mechanism producing advice (AI model, ruleset)
  Policy            declarative preference or constraint
  Metric            quantitative observation with baseline
  View              derived, read-only synthesis of events

(see CORE-CONCEPTS.txt)

------------------------------------------------------------

6. UCC CONTRACT LIFECYCLE
------------------------------------------------------------

Every UCC interaction follows this exact sequence:

  0. Evaluate preconditions (if declared)   → outcome=failed, failure_class=permanent if not satisfied
  1. Observe current state                  → O(W) = S0, or ? if indeterminate, or ⊗ if failed
  2. Evaluate declaration against S0        → S1 = desired state, or ⊥
  3. Compute diff                           → D = Δ(S0, S1)
  4. Decide and apply transition (if D ≠ ∅ and not inhibited)
  5. Report result

Result model (closed, UCC/2.0 — three-field model):

  observation field (always present):
    ok            — observation succeeded; see outcome
    indeterminate — observation ran but state is unclear
    failed        — observation infrastructure could not execute

  outcome field (when observation = ok):
    converged     — already correct, no action taken
    changed       — successfully transitioned (see completion)
    unchanged     — diff exists but transition deliberately not applied (see inhibitor)
    failed        — could not converge (see failure_class)

  completion field (when outcome = changed):
    complete      — transition verified synchronously
    pending       — transition initiated, awaiting verification cycle (mode=verify)

  failure_class field (when outcome = failed):
    retryable     — transient failure, retry may help
    permanent     — structural impossibility, retry futile

  inhibitor field (when outcome = unchanged):
    dry_run | whatif | policy

  partial field (optional, when outcome = changed):
    true          — multi-target: some sub-targets succeeded, some failed

(see UCC-EXECUTION.txt for normative spec, UCC-STATUS.txt for full result field semantics)

8. READING ORDER
------------------------------------------------------------

To understand the framework:
  1. CORE-CONCEPTS.txt      (canonical vocabulary for all concepts used across the framework)
  2. GLOSSARY.txt           (canonical terms)
  3. BISS.txt               (two axes)
  4. UCC-EXECUTION.txt      (normative spec)
  5. UCC-STATUS.txt         (status semantics)

To implement a UCC component:
  1. UCC-EXECUTION.txt      (normative spec + execution model in §4)
  2. UCC-CHECKLIST.txt      (compliance checklist)
  3. UCC-TESTS.txt          (test requirements)
  4. GOVERNED-WRAPPER-IMPLEMENTATIONS.txt
  5. UCC-FORMAL.txt         (mathematical foundations; read after UCC-EXECUTION.txt if formal proof is needed)

To use UCC/2.0 extensions:
  1. UCC-DIFF.txt             (diff equality contract — if customizing equality)
  2. UCC-EXT-CONCURRENCY.txt  (concurrency control — if concurrent writes exist)
  3. UCC-EXT-ASYNC.txt        (async convergence — if transitions are deferred)
  4. UCC-EXT-COMPENSATION.txt (compensation — if rollback semantics are needed)
  5. UCC-COMPOSITION.txt      (combination rules — if using more than one extension)

To apply UCC across multiple scopes (governance stack):
  1. UCC-SCOPE.txt            (scope taxonomy, caused_by, per-scope semantics)
  2. RECORD-MODEL-SPEC.txt    (record types produced at each scope)
  3. SYSTEMIC-GOVERNANCE-PROBLEMS.txt  (problem-to-scope mapping)

To evaluate fit:
  1. UCC-GUIDE-FIT-1.txt
  2. UCC-EXAMPLE.txt
