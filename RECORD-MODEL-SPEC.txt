RECORD-MODEL-SPEC.txt
===================

STATUS
------
Normative specification

VERSION
-------
1.0

SCOPE
-----
This document formally specifies the RECORD MODEL.

The record model is the minimal, sufficient substrate that addresses the
seven systemic problems identified in distributed, automated, and
AI-augmented systems.

This specification defines:
- what must be recorded
- how records relate
- what guarantees are provided

It explicitly does NOT define:
- execution logic
- control flow
- orchestration
- optimization
- prevention mechanisms

------------------------------------------------------------

CLASSIFICATION
--------------
Supporting – Immutable Record Model

------------------------------------------------------------


====================================================================
1. DEFINITIONS
====================================================================

RECORD
------
An immutable, append-only factual statement describing something that
actually happened.

Once persisted, a record:
- MUST NOT be modified
- MUST NOT be deleted
- MUST NOT be reinterpreted

ARTIFACT
--------
A persistent, identifiable object whose meaning must remain stable over
time. All records are artifacts.


====================================================================
2. CORE INVARIANTS
====================================================================

I1. Records are append-only  
I2. Records never change meaning  
I3. Records describe facts, not intentions or expectations  
I4. Execution MUST NOT depend on records  
I5. Removing the record layer MUST NOT change behavior  

Violation of any invariant invalidates this model.


====================================================================
3. MINIMAL RECORD (MANDATORY)
====================================================================

Every record MUST contain the following fields.

record:
  event_id           (string, globally unique)
  type               (string)
  correlation_id     (string)
  timestamp_utc      (RFC3339)

Field semantics:

event_id
- Identifies exactly one factual occurrence
- MUST be unique across all records

type
- Classifies the kind of fact
- Examples: intent, execution, decision, approval, quality, view

correlation_id
- Groups records belonging to the same logical intent
- MUST NOT be used to infer causality

timestamp_utc
- Time at which the fact was recorded
- Used only for ordering and inspection


====================================================================
4. CAUSALITY (OPTIONAL BUT EXPLICIT)
====================================================================

A record MAY include:

parent_event_id

Rules:
- At most one parent
- Parent MUST reference an existing event_id
- Absence of parent means "no declared causality"

Causality is never inferred.


====================================================================
5. RECORD TYPES
====================================================================

5.1 INTENT RECORD
-----------------
Purpose:
Declare that a logical operation exists.

Additional fields:
  description (string)

Semantics:
- Defines WHY something may happen
- Does not imply that anything actually happened


5.2 EXECUTION RECORD (BISS-RIG COMPATIBLE)
-----------------------------------------
Purpose:
Declare that an action actually ran.

Additional fields:
  action        (string)
  result        (structured as UCC three-field model: observation + outcome + qualifiers;
                 or 'success' | 'failure' for non-UCC interactions)
  final_state   (string)

Note: For non-UCC interactions, result MAY be represented as 'success' or 'failure'.
For UCC-compliant interactions, the three-field result model (observation/outcome/completion
with applicable qualifiers) MUST be used. See UCC-STATUS.txt for the complete taxonomy.

Semantics:
- The action executed
- The result is factual
- The final state is declared and observable


5.3 STATE DECLARATION RECORD
----------------------------
Purpose:
Make state interaction explicit.

Additional fields:
  state:
    reads    (list of strings)
    writes   (list of strings)
    assumes  (list of strings)

Semantics:
- Descriptive only
- No validation
- No enforcement
- Undeclared state usage is an observability defect


5.4 DECISION / REASONING RECORD
-------------------------------
Purpose:
Declare that a choice influenced execution.

Additional fields:
  reasoning_id        (string)
  reasoning_version   (string)
  outcome             (string)

Semantics:
- A choice was made
- Which logic produced it is explicit
- Reasoning is versioned and auditable


5.5 POLICY / AUTHORITY RECORD
-----------------------------
Purpose:
Declare that authority or priority was applied.

Additional fields:
  policy_id   (string)
  version     (string)
  actor       (string)

Semantics:
- Responsibility is explicit
- Priority is observable
- No enforcement implied


5.6 QUALITY RECORD
------------------
Purpose:
Observe quality independently of correctness.

Additional fields:
  metric:
    metric_id   (string)
    value       (number)
    baseline    (string)
    status      (normal | degraded | critical)

Semantics:
- Correctness and quality are orthogonal
- Quality never redefines success or failure


5.7 VIEW RECORD (DERIVED)
------------------------
Purpose:
Provide systemic visibility.

Additional fields:
  view_id
  source_events
  derivation_rules

Rules:
- Derived only
- Read-only
- Discardable
- MUST NOT influence execution


====================================================================
6. PROBLEM-TO-RECORD MAPPING (NORMATIVE)
====================================================================

Problem 1 – Governance of execution
→ Execution record

Problem 2 – Loss of explicit causality
→ parent_event_id + correlation_id

Problem 3 – Implicit state
→ State declaration record

Problem 4 – Ungoverned reasoning
→ Decision / reasoning record

Problem 5 – Compliance without quality
→ Quality record

Problem 6 – Absence of systemic visibility
→ View record

Problem 7 – Lack of global arbitration
→ Policy / authority record


====================================================================
7. NON-GOALS (EXPLICIT)
====================================================================

The record model MUST NOT:
- prevent execution
- enforce correctness
- infer missing data
- retry or rollback actions
- optimize behavior
- decide actions

Any system using records to control execution violates this spec.


====================================================================
8. EVOLVABILITY GUARANTEES
====================================================================

- Records are immutable facts
- Meaning is preserved forever
- New record types may be added
- Existing records remain valid
- Past data never changes meaning


====================================================================
9. COMPLIANCE CHECK
====================================================================

A system conforms to this specification if and only if:

- All records satisfy Section 3
- No record is mutable
- Causality is explicit or absent
- Execution does not depend on records
- Removing the record layer preserves behavior


====================================================================
10. FINAL STATEMENT
====================================================================

The record model does not prevent failure.
It prevents ambiguity.

By making all relevant facts explicit and immutable, the system becomes:
- auditable
- debuggable
- evolvable
- explainable

END OF SPECIFICATION
