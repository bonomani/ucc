1. INTRODUCTION

The Universal Convergence Contract (UCC) defines a universal, language-agnostic,
and protocol-independent contract for communication between software components.

UCC is based on a single principle:

All software interactions are attempts to converge an observed state
toward a desired state, with a finite and observable outcome.

------------------------------------------------------------

2. TERMINOLOGY

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT",
"SHOULD", "SHOULD NOT", "RECOMMENDED", and "MAY" in this document
are to be interpreted as described in RFC 2119.

State:
A finite, observable, serializable, and comparable representation
of the world.

Declaration:
A declarative specification of a desired target state.
NOTE: Not to be confused with Intent (correlation grouping) in CORE-CONCEPTS.

Observation:
The act of reading the current observable state.

Diff:
A pure comparison between two states.

Result:
The final observable outcome of a UCC transaction, expressed through
three orthogonal fields: observation, outcome, and completion (with
further qualifiers as applicable).

------------------------------------------------------------

3. OVERVIEW OF THE MODEL

Each UCC transaction consists of:

0. Precondition evaluation (if declared)
1. Observation of the current state
2. Evaluation of the declaration (skipped when mode=verify)
3. Computation of a state difference
4. Determination of a final result
5. Conditional application of a transition (only if D≠∅ and outcome≠unchanged;
   MUST NOT execute when mode=verify or mode=dry_run)
6. Result reporting

No step may be omitted or reordered.

------------------------------------------------------------

4. MESSAGE TYPES

UCC defines exactly three message types:

- DeclarationMessage       (standard declaration)
- CompensationDeclarationMessage  (compensation declaration, identified by `compensates`)
- ResultMessage

No other message types are permitted.

------------------------------------------------------------

5. DECLARATION MESSAGE

A DeclarationMessage MUST contain:

- meta
- declaration

The declaration section MUST specify:
- a target
- a desired_state

Optional fields in the declaration:
- mode         (apply | verify | dry_run; defaults to apply)
- requires     (preconditions)
- equality     (diff equality semantics — see UCC-DIFF.txt)
- concurrency  (concurrency control — see UCC-EXT-CONCURRENCY.txt)

A CompensationDeclarationMessage MUST contain:
- meta
- declaration with `compensates` field identifying the primary declaration

The compensation declaration MUST also specify:
- target, desired_state, trigger

Declarations MUST be declarative.
They MUST NOT encode actions or procedures.

------------------------------------------------------------

6. RESULT MESSAGE

A ResultMessage MUST contain:

- meta
- observe
- result

The observe section fields are result-conditional:
- observed_before and diff MUST be present when observation = ok.
- observed_before MUST be present (observed_after absent) when
  outcome=changed and completion=pending.
- diff MUST be present when outcome=changed and completion=pending.
- observed_after MUST be present when outcome=changed and completion=complete.
- observed_before, observed_after, and diff MUST be absent when
  observation = failed or indeterminate.

The result section MUST contain:
- observation (always)
- outcome (when observation = ok)
- completion (when outcome = changed)
- failure_class (when outcome = failed)
- inhibitor (when outcome = unchanged)

------------------------------------------------------------

7. RESULT MODEL

The result is expressed through three orthogonal fields:

observation (always present):
  ok            — observation succeeded; see outcome
  indeterminate — observation ran but state is unclear
  failed        — observation infrastructure could not execute

outcome (when observation = ok):
  converged     — already correct, no action taken
  changed       — successfully transitioned (see completion)
  unchanged     — diff exists, transition deliberately not applied (see inhibitor)
  failed        — could not converge (see failure_class)

completion (when outcome = changed):
  complete      — transition verified synchronously
  pending       — transition initiated, awaiting verification cycle (mode=verify)

failure_class (when outcome = failed):
  retryable     — transient failure, retry may help
  permanent     — structural impossibility, retry futile

inhibitor (when outcome = unchanged):
  dry_run       — suppressed by mode=dry_run
  whatif        — suppressed by whatif mode
  policy        — suppressed by policy constraint

partial (optional qualifier when outcome = changed):
  true          — multi-target: some sub-targets succeeded, some failed
  false         — all sub-targets succeeded (or single-target)

No additional field values are allowed.

See UCC-STATUS.txt for full semantics and consumer obligations.

------------------------------------------------------------

8. EXECUTION SEMANTICS

Given an observed state S0:

- If observation infrastructure failed (⊗):
    observation MUST be failed
- If observation ran but state is indeterminate (?):
    observation MUST be indeterminate
- When both are possible, MUST use observation=failed (see UCC-STATUS.txt §3)
- If precondition not satisfied:
    observation=ok, outcome MUST be failed, failure_class MUST be permanent
- If declaration evaluation returns ⊥ and permanent:
    observation=ok, outcome MUST be failed, failure_class MUST be permanent
- If declaration evaluation returns ⊥ and transient:
    observation=ok, outcome MUST be failed, failure_class MUST be retryable
- If diff(S0, S1) is empty:
    observation=ok, outcome MUST be converged
- If diff(S0, S1) is non-empty and transition not applied:
    observation=ok, outcome MUST be unchanged, inhibitor MUST be present
- If diff(S0, S1) is non-empty, transition initiated, not yet verifiable:
    observation=ok, outcome MUST be changed, completion MUST be pending
- If diff(S0, S1) is non-empty and transition applied with partial failure:
    observation=ok, outcome MUST be changed, completion MUST be complete, partial MUST be true
- If diff(S0, S1) is non-empty and transition applied successfully:
    observation=ok, outcome MUST be changed, completion MUST be complete

------------------------------------------------------------

9. IDEMPOTENCE

Applying the same declaration multiple times MUST produce
the same observable result.

Formally:
apply(declaration, apply(declaration, S)) = apply(declaration, S)

------------------------------------------------------------

10. ERROR HANDLING

Errors MUST NOT be represented as exceptions.

Errors MUST be reported using:
- observation=ok, outcome=failed, failure_class=retryable   (transient)
- observation=ok, outcome=failed, failure_class=permanent   (structural)
- observation=failed                                        (observation infrastructure failed)
- observation=indeterminate                                 (observation ran, state indeterminate)

All failures MUST include a non-empty message.

------------------------------------------------------------

11. OBSERVATION REQUIREMENTS

Observation MUST:

- precede any state change
- be read-only
- be deterministic
- reflect actual observable state

Skipping observation is non-compliant.

------------------------------------------------------------

12. TRANSPORT INDEPENDENCE

UCC does not define transport mechanisms.

Any transport MAY be used, provided it does not alter
the semantics defined in this document.

------------------------------------------------------------

13. SECURITY CONSIDERATIONS

UCC defines no authentication, authorization, or encryption mechanisms.

Security concerns MUST be addressed by the transport
or surrounding system.

------------------------------------------------------------

14. COMPLIANCE

An implementation is UCC/2.0 compliant if and only if:

- it conforms to UCC-EXECUTION.txt (authoritative)
- it emits valid UCC/2.0 messages
- it respects all execution semantics in Section 8
- it passes all eight mandatory test phases (see UCC-TESTS.txt for the complete list of mandatory phases)
- it complies with any declared extension blocks (R13–R16)

------------------------------------------------------------

15. IANA CONSIDERATIONS

This document has no IANA considerations.

------------------------------------------------------------

16. CONCLUSION

UCC defines a minimal, deterministic, and universal contract
for software interaction.

It removes language, runtime, and protocol dependencies,
leaving only observable state, declaration, and convergence.
