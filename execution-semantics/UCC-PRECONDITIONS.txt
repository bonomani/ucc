UCC-PRECONDITIONS – Declaration Precondition Model
Universal Convergence Contract (UCC)

This document is NORMATIVE.
It defines how dependencies between UCC declarations MUST be declared,
evaluated, and enforced.

------------------------------------------------------------

PREREQUISITE READING
--------------------
UCC-EXECUTION.txt   (core execution model — mandatory)
GLOSSARY.txt        (canonical vocabulary)

TIER

Tier 1 – Core. Companion to UCC-EXECUTION.txt. Defines Step 0 precondition model.

------------------------------------------------------------

1. PURPOSE OF THIS DOCUMENT

UCC contracts are designed to be independently idempotent.
However, in practice, some declarations cannot reach the desired state
unless another declaration has already converged.

Example:
  Installing a browser extension requires the browser to be installed first.
  Configuring a service requires the service to exist first.

Without an explicit model for these dependencies, implicit ordering
and hidden preconditions reintroduce the problems UCC was designed to eliminate.

This document defines:

- what a precondition is
- how preconditions are declared
- how preconditions are evaluated
- what happens when a precondition is not satisfied

------------------------------------------------------------

2. DEFINITIONS

Precondition
A declared dependency on the converged state of another declaration.

Precondition Set
The complete set of preconditions for a given declaration.

Precondition Evaluation
The act of observing whether all declared preconditions are satisfied
before executing the declaration.

------------------------------------------------------------

3. CORE PRINCIPLE

Preconditions are DECLARATIVE, not procedural.

A precondition does NOT say: "execute this declaration first."
A precondition says: "this state must already be true."

The execution order is a consequence of the dependency graph,
not an imperative instruction.

------------------------------------------------------------

4. PRECONDITION DECLARATION

A declaration MAY include preconditions in its declaration message.

4.1 Declaration Schema with Preconditions

{
  "meta": {
    "contract": "string",
    "version": "2.0",
    "id": "string",
    "timestamp": "ISO-8601"
  },
  "declaration": {
    "target": "string",
    "desired_state": {},
    "requires": [
      {
        "declaration_id": "string",
        "target": "string",
        "desired_state": {}
      }
    ]
  }
}

Notes:
- requires is OPTIONAL
- Each entry in requires is a minimal state assertion
- The declared desired_state in requires is the minimum state that must be true
- requires does NOT trigger execution of the referenced declaration
- requires only declares an observation-time check

------------------------------------------------------------

5. PRECONDITION EVALUATION

Before any transition is attempted, the component MUST:

1. Evaluate all declared preconditions
2. Observe the state of each required target
3. Verify that each observed state satisfies the declared desired_state

If all preconditions are satisfied: proceed normally.
If any precondition is not satisfied: result MUST be observation=ok,
outcome=failed, failure_class=permanent with a message identifying the
unsatisfied precondition.
If a precondition observation cannot execute: result MUST be
observation=failed with Step 0 evidence identifying the failed check.
If a precondition observation executes but is indeterminate: result MUST
be observation=indeterminate with Step 0 evidence identifying the
indeterminate check.

Rationale for failure_class=permanent:
An unsatisfied precondition is a structural gap, not a transient condition.
Retrying the same declaration will not satisfy the missing precondition.
A different declaration must be executed first.

------------------------------------------------------------

6. EXECUTION SEQUENCE WITH PRECONDITIONS

The extended execution sequence is:

Step 0 – Precondition evaluation
  For each declared precondition:
    Observe the required target state
    If observation cannot execute (⊗) → observation=failed
    If observation is indeterminate (?) → observation=indeterminate
    If state does not satisfy required desired_state
      → observation=ok, outcome=failed, failure_class=permanent

Step 1 – Observe current state of primary target
Step 2 – Evaluate declaration into desired state
Step 3 – Compute diff
Step 4 – Decide whether execution should be attempted
Step 5 – Attempt transition (if applicable)
Step 6 – Verify, classify, and report result

Step 0 MUST occur before Step 1.
If Step 0 produces observation=failed, observation=indeterminate, or
outcome=failed, failure_class=permanent, Steps 1–6 MUST NOT execute.

------------------------------------------------------------

7. DEPENDENCY GRAPH

When multiple declarations declare preconditions on each other,
they form a dependency graph.

Rules for the dependency graph:

PREC-R1 – No cycles
A dependency graph MUST be a directed acyclic graph (DAG).
Circular dependencies are forbidden and constitute a design error.

PREC-R2 – Explicit only
Dependencies MUST NOT be inferred from execution order.
All dependencies MUST be declared in the declaration.

PREC-R3 – Minimal
Each precondition MUST declare only the minimum required state.
Over-specifying preconditions creates unnecessary fragility.

PREC-R4 – Observable
Every declared precondition MUST be observable.
A precondition on a non-observable state is forbidden.

------------------------------------------------------------

8. ORCHESTRATION IMPLICATIONS

An orchestrator that handles multiple declarations SHOULD:

1. Build the dependency graph from declared preconditions
2. Execute declarations in topological order
3. Halt execution of dependent declarations if a precondition fails
4. Report the dependency failure clearly

An orchestrator MUST NOT:
- Reorder declarations without respecting the dependency graph
- Execute a dependent declaration when its precondition declaration has failed
- Assume implicit ordering from declaration order

------------------------------------------------------------

9. PRECONDITION FAILURE SEMANTICS

When a precondition is not satisfied:

result:  observation=ok, outcome=failed, failure_class=permanent
message: MUST identify:
         - which precondition failed
         - which declaration_id was required
         - what state was expected
         - what state was observed
observe.preconditions:
         REQUIRED
         MUST contain a Step 0 report with:
         - evaluated=true
         - satisfied=false
         - at least one check with observation=ok and satisfied=false

Example message:
  "Precondition not satisfied: declaration 'ensure-firefox' (target: firefox)
   requires installed=true but observed installed=false"

When a precondition observation fails:

result:  observation=failed
message: MUST identify which precondition check could not be observed
observe.preconditions:
         REQUIRED
         MUST contain a Step 0 report with at least one check whose
         observation=failed and satisfied=false

When a precondition observation is indeterminate:

result:  observation=indeterminate
message: MUST identify which precondition check was indeterminate
observe.preconditions:
         REQUIRED
         MUST contain a Step 0 report with at least one check whose
         observation=indeterminate and satisfied=false

------------------------------------------------------------

10. WHAT PRECONDITIONS ARE NOT

Preconditions are NOT:
- execution triggers ("run X then run Y")
- pipeline steps ("if X then Y")
- retry triggers ("wait for X to converge")
- implicit assumptions ("X must have run before")

Preconditions are ONLY:
- observable state assertions evaluated at execution time

------------------------------------------------------------

11. RELATION TO COMPOSABILITY

UCC/2.0 Section 8 guarantees composability.
This document makes composability explicit and verifiable.

Without precondition declarations:
- composability is assumed but not checkable
- failures are discovered at runtime, not at design time

With precondition declarations:
- composability is declared and observable
- dependency violations are detectable before execution
- the dependency graph is auditable

------------------------------------------------------------

12. COMPLIANCE CRITERION

An implementation is precondition-compliant if and only if:

- All preconditions are declared in the declaration schema
- Precondition evaluation occurs before any transition
- Unsatisfied preconditions produce outcome=failed, failure_class=permanent
- Failed precondition observations produce observation=failed
- Indeterminate precondition observations produce observation=indeterminate
- The dependency graph is acyclic
- Precondition failures are reported with full identification

------------------------------------------------------------

13. NORMATIVE SUMMARY

Preconditions are declarative state assertions on other declarations.

They are evaluated before any transition.
They never trigger execution.
They produce outcome=failed, failure_class=permanent when unsatisfied.
They propagate observation=failed or observation=indeterminate when the
precondition checks themselves cannot establish a determinate state.
They form a verifiable, auditable dependency graph.

Explicit preconditions eliminate hidden dependencies
and make composability observable.
