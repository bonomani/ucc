UCC-EXECUTION – Core Execution Contract
========================================

TIER
-----
Tier 1 – Core Execution.
Depends on: UCC-OBSERVATION, UCC-DESIRED-STATE, UCC-DIFF.

VERSION
-------
1.0

NORMATIVE DOCUMENT
Any compliant implementation MUST respect all rules defined below.
In case of conflict with any other UCC document, this document is
authoritative for the core execution model (R1–R12).

NAMING NOTE
-----------
This document uses "declaration" for the desired-state specification.
This avoids collision with "Intent" (correlation grouping of events)
defined in CORE-CONCEPTS. The two terms are unrelated. See GLOSSARY.txt
for canonical definitions.

------------------------------------------------------------

1. SCOPE
------------------------------------------------------------

UCC/1.0 applies to any software interaction involving:

- files
- registry
- databases
- services
- APIs
- scripts
- agents
- distributed systems

The contract describes observable state transitions, not imperative
calls or internal implementations.

This document governs the core execution lifecycle. The three Tier 0
primitives that underpin it are:

- UCC-OBSERVATION  — how current state is read from the world
- UCC-DESIRED-STATE — how desired state is specified
- UCC-DIFF         — how the difference between observed and desired
                     state is computed

The corresponding abstract primitives are defined in
../contract-semantics/UCC-DESIRED-STATE.txt,
../contract-semantics/UCC-DIFF.txt, and
../contract-semantics/UCC-OBSERVATION.txt.

Extension contracts (equality semantics, concurrency control, async
deferred convergence, compensation) are defined in separate Tier 2
documents and are optional unless the corresponding declaration block
is used.

Scope extension (Tier 1): UCC-SCOPE.txt extends this contract to five
execution scopes (operation, decision, system, policy, intent) and
defines cross-scope causality via the caused_by field in meta. Scope
defaults to "operation" when absent; this document remains fully
authoritative for all operation-scope behavior.

------------------------------------------------------------

2. RESULT MODEL
------------------------------------------------------------

The result is expressed through three orthogonal fields: `observation`,
`outcome`, and `completion`. Additional qualifiers (`failure_class`,
`inhibitor`, `partial`) carry further detail.

observation field (always present):
  ok            — the observation infrastructure ran and produced a determinate state
  indeterminate — observation ran but could not produce a determinate state
  failed        — observation infrastructure did not run

outcome field (present only when observation = ok):
  converged     — diff is empty; no transition was needed
  changed       — a transition was executed or successfully initiated in this cycle
  unchanged     — diff is non-empty but transition was deliberately not applied
  failed        — transition failed

completion field (present only when outcome = changed):
  complete      — the transition result is verifiable now
  pending       — the transition has been initiated but cannot yet be verified

failure_class field (present only when outcome = failed):
  retryable     — failure is transient; retrying may succeed
  permanent     — failure is structural; retrying will not help

inhibitor field (present only when outcome = unchanged):
  dry_run       — transition not applied due to dry-run mode
  whatif        — transition not applied due to whatif mode
  policy        — transition not applied due to a policy constraint

partial field (present only when outcome = changed; defaults to false):
  true          — at least one sub-target succeeded, at least one failed
  false         — all sub-targets reached the desired state

No other field values exist.

Field presence rules:

  observation
    ALWAYS present.

  outcome
    REQUIRED when observation = ok.
    MUST be omitted when observation = indeterminate or failed.

  completion
    REQUIRED when outcome = changed.
    MUST be omitted otherwise.

  failure_class
    REQUIRED when outcome = failed.
    MUST be omitted otherwise.

  inhibitor
    REQUIRED when outcome = unchanged (R11).
    MUST be omitted otherwise.

  partial
    OPTIONAL when outcome = changed; defaults to false.
    MUST be omitted when outcome ≠ changed.

  partial_detail
    REQUIRED when partial = true.
    MUST be omitted when partial = false or absent.

  observe.preconditions
    OPTIONAL when no preconditions are declared.
    MAY be present when Step 0 completed successfully.
    REQUIRED when Step 0 blocks execution or when a Step 0
    observation failed or became indeterminate.

  observed_before
    REQUIRED when Step 1 completed successfully.
    MUST be absent when Step 0 blocks execution or when
    observation = indeterminate or failed.

  diff
    REQUIRED when Step 3 completed successfully.
    MUST be absent when Step 0 blocks execution, when
    observation = indeterminate or failed, or when declaration
    evaluation returned ⊥ before diff could be computed.

  observed_after
    REQUIRED when outcome = changed AND completion = complete.
    MUST be a post-mutation observation of the target state.
    MUST be absent when observation ≠ ok, outcome ≠ changed, or
    completion = pending.

  proof
    REQUIRED when outcome = changed (R10).
    OPTIONAL otherwise.
    NOTE: UCC-EXT-EQUALITY adds a further proof obligation when
    equality.mode = semantic and outcome = converged.

  message
    REQUIRED when outcome = failed or observation ≠ ok.
    REQUIRED when outcome = unchanged (R11).

Canonical result paths (closed set):

  PF-1 Precondition unsatisfied
    result: observation=ok, outcome=failed, failure_class=permanent
    observe.preconditions: REQUIRED with at least one unsatisfied check
    observed_before / diff / observed_after: absent

  PF-2 Precondition observation failed
    result: observation=failed
    observe.preconditions: REQUIRED and MUST identify the failed check
    observed_before / diff / observed_after / outcome: absent

  PF-3 Precondition observation indeterminate
    result: observation=indeterminate
    observe.preconditions: REQUIRED and MUST identify the indeterminate check
    observed_before / diff / observed_after / outcome: absent

  O-1 Primary observation failed
    result: observation=failed
    observed_before / diff / observed_after / outcome: absent

  O-2 Primary observation indeterminate
    result: observation=indeterminate
    observed_before / diff / observed_after / outcome: absent

  E-1 Declaration evaluation impossible
    result: observation=ok, outcome=failed, failure_class=retryable|permanent
    observed_before: REQUIRED
    diff / observed_after: absent

  D-1 Already converged
    result: observation=ok, outcome=converged
    observed_before / diff: REQUIRED
    diff MUST be empty
    observed_after: absent

  D-2 Transition inhibited
    result: observation=ok, outcome=unchanged, inhibitor=<reason>
    observed_before / diff: REQUIRED
    diff MUST be non-empty
    observed_after: absent

  T-1 Transition changed and verified
    result: observation=ok, outcome=changed, completion=complete
    observed_before / diff / observed_after / proof: REQUIRED

  T-2 Transition changed and pending verification
    result: observation=ok, outcome=changed, completion=pending
    observed_before / diff / proof / verification: REQUIRED
    observed_after: absent

  T-3 Transition partial
    result: observation=ok, outcome=changed, completion=complete, partial=true
    observed_before / diff / observed_after / proof / partial_detail: REQUIRED

  T-4 Transition failed after attempt
    result: observation=ok, outcome=failed, failure_class=retryable|permanent
    observed_before / diff: REQUIRED
    observed_after: absent

3. CANONICAL MESSAGE STRUCTURE
------------------------------------------------------------

3.1 Declaration message (input)

A declaration message — Form A (standard declaration):

{
  "meta": {
    "contract": "string",
    "version": "1.0",
    "id": "string",
    "timestamp": "ISO-8601"
  },
  "declaration": {
    "mode": "apply | verify | dry_run",
    "target": "string",
    "desired_state": {},
    "requires": [],
    "verification_token": "string"
  }
}

Notes:
- mode defaults to "apply" when absent.
- mode = "verify" requires a verification_token field
  (see UCC-EXT-ASYNC for the complete verification lifecycle).
- requires is OPTIONAL. Each entry is a minimal state assertion
  evaluated before Step 1. See UCC-PRECONDITIONS for the full
  precondition model.
- The `equality` block (volatile fields, semantic mode, array order
  override) is an OPTIONAL extension defined in UCC-EXT-EQUALITY.
- The `concurrency` block (optimistic/pessimistic strategy) is an
  OPTIONAL extension defined in UCC-EXT-CONCURRENCY.
- Compensation declarations (identified by `compensates`) are defined
  in UCC-EXT-COMPENSATION.

------------------------------------------------------------

3.2 Result message (output)

{
  "meta": {
    "contract": "string",
    "version": "1.0",
    "id": "string",
    "timestamp": "ISO-8601",
    "duration_ms": "integer"
  },
  "observe": {
    "preconditions": {
      "evaluated": true,
      "satisfied": true,
      "checks": []
    },
    "observed_before": {},
    "observed_after": {},
    "diff": {}
  },
  "result": {
    "observation": "ok | indeterminate | failed",
    "outcome":     "converged | changed | unchanged | failed",
    "completion":  "complete | pending",
    "failure_class": "retryable | permanent",
    "inhibitor":   "dry_run | whatif | policy",
    "partial":     true,
    "partial_detail": {
      "total": "integer",
      "converged": "integer",
      "changed": "integer",
      "failed": "integer"
    },
    "proof": {},
    "message": "string"
  }
}

NOTE: Extension blocks (verification, compensation, compensation_available,
and equality-dependent proof fields) are defined in the respective
UCC-EXT-* documents and are present in the result only when the
corresponding extension is used.

Step 0 evidence is carried in `observe.preconditions` when preconditions
are declared. Each check records the required declaration, the expected
state, the observed state when available, and whether observation of the
precondition itself succeeded.

------------------------------------------------------------

4. EXECUTION MODEL
------------------------------------------------------------

A UCC interaction is a single, atomic evaluation cycle.
No step may be skipped. No step may be reordered.

Step 0 – Precondition evaluation (if declared)
  Observe every declared precondition target before Step 1.
  If any precondition observation returns ⊗
    → emit PF-2, STOP.
  If any precondition observation returns ?
    → emit PF-3, STOP.
  If any precondition is not satisfied
    → emit PF-1, STOP.
  Step 0 is a hard gate. If any precondition fails, the cycle MUST
  terminate immediately. Step 1 MUST NOT be executed.
  See UCC-PRECONDITIONS for the full precondition model.

Step 1 – Observation
  S0 = observe(W)
  If O(W) = ⊗ → observation=failed, STOP
  If O(W) = ?  → observation=indeterminate, STOP
  MUST be read-only, deterministic, complete.
  (Governed by UCC-OBSERVATION.)

Step 2 – Declaration evaluation
  S1 = evaluate(declaration, S0)
  MUST be declarative. MUST NOT perform side effects.
  Returns S1 (desired state) or ⊥ (impossible).
  When mode = verify: this step is skipped; S1 is recovered from the
  verification token. (Governed by UCC-DESIRED-STATE.)
  When mode = verify: see UCC-EXT-ASYNC for the full two-phase lifecycle.

Step 3 – Diff computation
  D = diff(S0, S1)
  Equality semantics are governed by UCC-DIFF (defaults) and
  UCC-EXT-EQUALITY (if an `equality` block is declared).
  MUST be a pure function. D = ∅ if and only if S0 ≡ S1.
  Step 3 MUST NOT execute when Step 2 returned ⊥.

Step 4 – Execution decision (evaluated in order, first match wins):

  -- Step 0 gate already ensures preconditions are satisfied.
  If S1 = ⊥ and failure is permanent       → final result path = E-1 with failure_class=permanent
  If S1 = ⊥ and failure is transient       → final result path = E-1 with failure_class=retryable
  If D = ∅                                  → final result path = D-1
  If D ≠ ∅ and transition is inhibited     → final result path = D-2 with inhibitor=<reason>
  If D ≠ ∅ and no inhibitor applies        → proceed to Step 5

Step 5 – Transition attempt
  Execute the mutation required to move from S0 toward S1.
  This is the only step that may mutate target state.
  Declaration evaluation and transition execution MUST remain distinct.
  If mode = verify or mode = dry_run: Step 5 MUST NOT execute.
  Concurrency control (if declared) is applied at this step
  immediately before any mutation. See UCC-EXT-CONCURRENCY.

COMPENSATION INTERCEPTION
  If a compensation declaration (UCC-EXT-COMPENSATION) is linked and
  its trigger condition is met, the compensation lifecycle executes
  after the primary result has been classified in Step 6 and before
  final emission. See UCC-EXT-COMPENSATION for the
  compensation execution model. The primary outcome is always preserved;
  compensation is reported in a separate `compensation` block alongside it.

Step 6 – Verification, classification, and reporting
  If no transition attempt was required, emit the final result selected
  in Step 4.

  If Step 5 executed and synchronous verification is possible, perform a
  post-mutation observation and classify from the observed result:
  - observed_after ≡ desired state under the active equality semantics
      → emit T-1
  - at least one sub-target reached desired state and at least one failed
      → emit T-3
  - transition failed before verified convergence
      → emit T-4 with failure_class=retryable|permanent

  If Step 5 executed and the result cannot yet be synchronously verified:
  - emit T-2
  - a verification token is REQUIRED
  - observed_after MUST be omitted

  Emit exactly one result object per Section 3.2.
  The reported result MUST match the actual behavior.
  If compensation was triggered, emit the compensation block (see
  UCC-EXT-COMPENSATION).

IDEMPOTENCE GUARANTEE
  evaluate(declaration, evaluate(declaration, S)) = evaluate(declaration, S)
  This guarantees safe retries, crash recovery, and distributed
  orchestration.

DRY-RUN SEMANTICS
  Dry-run is mode = dry_run. Step 0 and Step 1 behavior is unchanged.
  Only Step 5 is suppressed.
  - If Step 0 precondition is unsatisfied in dry_run mode, emit PF-1.
  - If Step 0 precondition observation fails, emit PF-2.
  - If Step 0 precondition observation is indeterminate, emit PF-3.
  - If Step 1 observation fails (O(W) = ⊗), emit O-1.
  - If Step 1 observation is indeterminate (O(W) = ?), emit O-2.
  - If D = ∅ in dry_run, emit D-1.
  - If D ≠ ∅ in dry_run, emit D-2 with inhibitor=dry_run.

5. NORMATIVE RULES
------------------------------------------------------------

R1 – Prior observation
Every transition MUST be preceded by an observation.

R2 – Declarativity
A declaration MUST NOT describe an action, only a target state.

R3 – Determinism
Given identical observation and declaration, the result MUST be identical.

R4 – Idempotence
Executing the same declaration twice MUST converge to the same final state.

R5 – Exhaustive result model
No observation, outcome, completion, failure_class, or inhibitor value
outside the defined sets is allowed. Every result MUST be expressed
through the three-field model (observation + outcome +
completion/failure_class/inhibitor as applicable).

R6 – No hidden state
All observable effects MUST be represented in the contract.

R7 – Observation indeterminacy
If the observation function runs but cannot produce a determinate state,
the result MUST be observation=indeterminate. No transition may be
attempted.

R8 – Partial convergence
If a target contains multiple sub-targets and at least one reached the
desired state while at least one failed, the result MUST be
outcome=changed with partial=true.
partial_detail MUST report the exact counts.

R9 – Failure classification
Any failure MUST set observation=ok, outcome=failed and MUST classify
failure_class as retryable or permanent.
Observation failures MUST set observation=failed.
Observation indeterminacy MUST set observation=indeterminate.
Unclassified failures are forbidden.

R10 – Proof obligation
A result with outcome=changed MUST include a non-empty `proof` object.
An empty or absent proof on a changed result is a contract violation.

R11 – Unchanged inhibition reason
A result with outcome=unchanged MUST carry an `inhibitor` field.
Valid inhibitors: dry_run, whatif, policy.
An unchanged result without a stated inhibitor is non-compliant.

R12 – Observation infrastructure failure
If the observation mechanism could not execute (⊗), the result MUST be
observation=failed. This is distinct from observation=indeterminate
(where observation ran but returned an indeterminate state).
When classification is ambiguous (e.g., a timeout where it is unclear
whether the request reached the target), the implementation MUST use
observation=failed. Defaulting to observation=indeterminate in ambiguous
cases is forbidden.
No transition may be attempted. No diff may be computed.

Extension rules R13–R16 are defined in UCC-EXT-EQUALITY,
UCC-EXT-CONCURRENCY, UCC-EXT-ASYNC, and UCC-EXT-COMPENSATION
respectively.

6. STATUS SEMANTICS
------------------------------------------------------------

observation = ok, outcome = converged
  The observed state was already equal to the desired state.
  No transition was applied. Safe to retry without effect.

observation = ok, outcome = changed, completion = complete
  A transition was executed and synchronously verified.
  observed_after is a post-mutation observation and equals the desired
  state under the active equality semantics. Proof MUST be present.

observation = ok, outcome = changed, completion = pending
  A transition has been initiated in Step 5 but the result cannot yet be
  verified in Step 6.
  A verification token MUST be present. observed_after MUST be omitted.
  The consumer MUST NOT assume convergence until verification completes.
  See UCC-EXT-ASYNC for the complete two-phase lifecycle.

observation = ok, outcome = changed, partial = true
  The target contains multiple sub-targets. At least one reached the
  desired state. At least one failed. partial_detail MUST be present.

observation = ok, outcome = unchanged
  The diff is non-empty but the transition was deliberately not applied.
  The `inhibitor` field MUST be present.

observation = ok, outcome = failed, failure_class = retryable
  The transition attempt could not produce verified convergence due to a
  transient condition.
  Retrying with the same declaration is expected to eventually succeed.

observation = ok, outcome = failed, failure_class = permanent
  Either declaration evaluation proved the target state structurally
  impossible before diff, a precondition was unsatisfied in Step 0, or
  the mutation attempt failed with a permanent condition and will never
  succeed.
  Retrying is futile. Human intervention may be required.

observation = failed
  The observation infrastructure could not execute.
  No diff was computed. No transition was attempted.
  When observation failure is ambiguous, this takes precedence over
  observation=indeterminate.

observation = indeterminate
  The observation ran but could not produce a determinate state.
  No transition was attempted. No diff is available.
  The consumer MUST NOT assume any state about the target.

------------------------------------------------------------

7. MINIMAL INTERFACE
------------------------------------------------------------

observe() → State | ? | ⊗
evaluate(declaration, State) → State | ⊥
diff(State, State) → Diff

Transition execution is required by Step 5, but its callable interface is
implementation-defined. It MUST remain distinct from declaration
evaluation and MUST NOT be hidden inside `evaluate()`.

8. GUARANTEED PROPERTIES
------------------------------------------------------------

- total language independence
- transport independence
- idempotence
- replayability
- auditability
- composability
- deterministic orchestration
- safe retry classification
- async convergence with verified proof
- compensation traceability

------------------------------------------------------------

9. EXPLICIT PROHIBITIONS
------------------------------------------------------------

Core prohibitions (this document):

- imperative calls
- unformalized exceptions
- implicit states
- runtime dependencies
- undocumented conventions
- non-observable "best effort" behavior
- unclassified failures
- proof-free changed results
- inhibitor-free unchanged results

Extension-specific prohibitions (see UCC-EXT-* and UCC-COMPOSITION):

- completion=pending without verification tokens
  (see UCC-EXT-ASYNC)
- pessimistic locking combined with completion=pending
  (see UCC-COMPOSITION.txt COMP-1)
- compensation_available without expires_at
  (see UCC-EXT-COMPENSATION)
- nested compensation declarations (compensates field on a compensation
  declaration) (see UCC-EXT-COMPENSATION and UCC-COMPOSITION COMP-2)
- verification warnings with codes outside the closed vocabulary
  (see UCC-EXT-ASYNC)
- expires_at exceeding Phase-1 timestamp + 24 hours when max_lifetime_ms
  is absent (see UCC-EXT-ASYNC)

------------------------------------------------------------

10. COMPLIANCE CRITERION
------------------------------------------------------------

An implementation is compliant with UCC-EXECUTION if R1–R12 are
respected and all interactions conform to the execution model defined
in Section 4, the result model in Section 2, and the canonical message
structure in Section 3.

Extension compliance is OPTIONAL unless the corresponding declaration
block is used. Once declared, compliance with the relevant UCC-EXT-*
document is MANDATORY.

------------------------------------------------------------

11. NORMATIVE SUMMARY
------------------------------------------------------------

Any valid software interaction is a declarative attempt to converge
an observed state toward a desired state, with a finite, deterministic,
classified, and observable result.

The result is expressed through three orthogonal fields: observation
(always), outcome (when observation succeeded), and completion or
failure_class or inhibitor (as applicable).

The core execution model is grounded in three Tier 0 primitives:
UCC-OBSERVATION, UCC-DESIRED-STATE, and UCC-DIFF. Each primitive defines
a pure, side-effect-free contract. The execution model composes them
into a single, atomic evaluation cycle (Steps 0–6).

Extensions to this contract (equality semantics, concurrency control,
async deferred convergence, and compensation) are optional but fully
normative when declared. Every declared extension creates binding
obligations on the implementation.
