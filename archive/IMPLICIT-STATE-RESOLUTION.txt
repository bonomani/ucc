IMPLICIT-STATE-RESOLUTION.txt
===========================

ADDRESSES
---------
Implicit State (see SYSTEMIC-GOVERNANCE-PROBLEMS.txt)

PURPOSE
-------
This document defines a minimal, concrete, and technology-agnostic method
to resolve **Problem #3: Implicit State and Hidden Side Effects** in
distributed, automated, or AI-augmented systems.

This approach complements BISS-RIG.
It does NOT modify BISS-RIG.
It makes state visible, not controlled.


PROBLEM STATEMENT
-----------------
Systems often read or modify state that is not explicitly declared:
- shared databases
- caches
- files
- registries
- environment variables
- global memory

As a result:
- components are implicitly coupled
- effects propagate non-locally
- failures are unpredictable
- reasoning about behavior is impossible

BISS-RIG observes final state.
It does NOT describe which state was relied upon.


CORE PRINCIPLE
--------------
State must be **declared**, not inferred.

State declaration:
- describes what is read
- describes what is written
- describes what is assumed invariant

It does NOT:
- enforce correctness
- block execution
- synchronize access
- decide behavior


MINIMAL SOLUTION
----------------
Introduce a **State Declaration Envelope** attached to each execution.

This envelope is declarative metadata.
Execution behavior remains unchanged.


STATE DECLARATION MODEL (MINIMAL)
--------------------------------

Each execution declares three sets:

reads:
- list of state identifiers that are read

writes:
- list of state identifiers that are modified

assumes:
- list of invariants assumed to be true


STATE IDENTIFIER
----------------
A state identifier is a logical name, not an implementation detail.

Examples:
- db.users.table
- cache.session_store
- file:/etc/app/config.yaml
- registry:HKLM\\Software\\MyApp
- env:PATH

The identifier must be:
- stable
- human-readable
- consistent across executions


EVENT STORE INTEGRATION
-----------------------
State declarations are recorded alongside execution events.

They are:
- append-only
- immutable
- descriptive only

They do NOT replace existing logs or metrics.


EVENT EXAMPLE
-------------

Execution event with state declaration:

event_id: EVT-42
correlation_id: CORR-9
parent_event_id: EVT-41
event_type: execution

state:
  reads:
    - db.inventory.items
    - cache.pricing_rules
  writes:
    - db.orders.table
  assumes:
    - invariant.inventory_non_negative


DECLARATION RULES
-----------------

Rule 1: No implicit state
- If state is used, it must be declared
- Undeclared state is a defect in observability

Rule 2: Reads and writes are distinct
- Reading a state does not imply writing it
- Writing must always be explicit

Rule 3: Assumptions are first-class
- Invariants must be declared explicitly
- Violated assumptions explain failures


WHAT THIS ENABLES
-----------------
With explicit state declarations, the system can:

- Detect hidden coupling between components
- Explain non-local failures
- Identify shared state hotspots
- Reconstruct side effects
- Perform accurate impact analysis

Example questions now answerable:
- Which executions touched this database table?
- Which components rely on this cache?
- What state was assumed when this failure occurred?


WHAT THIS DOES NOT DO
--------------------
This mechanism does NOT:
- validate state correctness
- lock state
- prevent races
- enforce isolation
- change execution behavior

State declaration is descriptive, not prescriptive.


MENTAL MODEL
------------
Execution does two things:
1. Acts (BISS-RIG)
2. Declares state interaction (this layer)

They must remain independent.


RELATION TO BISS-RIG
-------------------
- BISS-RIG proves final state
- State declaration explains dependencies and side effects

Both are required for reasoning.
They must not be merged.


SUCCESS CRITERIA
----------------
Problem #3 is resolved if and only if:

- Every execution declares state reads
- Every execution declares state writes
- Every execution declares critical assumptions
- No execution relies on undeclared state


FINAL NOTE
----------
Implicit state is the primary source of hidden coupling.

Making state explicit does not reduce power.
It reduces surprise.

Any attempt to infer state usage automatically
reintroduces ambiguity and must be avoided.
