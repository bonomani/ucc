UCC-IMPL-GUIDE – Minimal Implementation Guide
Universal Convergence Contract (UCC)

This document is NON-NORMATIVE but AUTHORITATIVE.
It explains the minimal requirements to implement UCC/2.0 correctly,
without prescribing any language, framework, or runtime.

------------------------------------------------------------

CLASSIFICATION
--------------
Guidance – Implementation Reference

------------------------------------------------------------

1. PURPOSE OF THIS DOCUMENT

This document explains:

- how to implement a UCC-compliant component
- what the absolute minimum logic is
- what can be ignored safely
- how to avoid common implementation mistakes

This guide is intentionally minimal.
Anything not required by UCC/2.0 is excluded.

------------------------------------------------------------

2. IMPLEMENTATION PHILOSOPHY

A UCC implementation is NOT:

- an API framework
- a workflow engine
- a message broker
- a state machine library

A UCC implementation IS:

- a deterministic evaluator of state convergence
- a translator between declaration and observable state
- a producer of verifiable results

------------------------------------------------------------

3. MINIMAL COMPONENT RESPONSIBILITIES

A UCC-compliant component MUST implement exactly three logical operations:

1. observe()
2. apply()
3. diff()

No other mandatory operations exist.

------------------------------------------------------------

4. REQUIRED FUNCTIONS (LANGUAGE-AGNOSTIC)

4.1 observe()

Purpose:
- read the current observable state

Requirements:
- MUST be read-only
- MUST be deterministic
- MUST return a finite serializable structure

Signature (conceptual):
observe() → State

------------------------------------------------------------

4.2 apply(declaration, state)

Purpose:
- evaluate the declaration against the observed state
- compute the desired target state

Requirements:
- MUST be declarative
- MUST NOT perform side effects
- MUST return either:
  - a desired state
  - or ⊥ if impossible

Signature (conceptual):
apply(Declaration, State) → State | ⊥

------------------------------------------------------------

4.3 diff(state_before, state_after)

Purpose:
- compute the difference between two states

Requirements:
- MUST be a pure function
- MUST satisfy:
  diff(a, b) = ∅ if and only if a ≡ b

Signature (conceptual):
diff(State, State) → Diff

------------------------------------------------------------

5. EXECUTION MODEL
------------------
The normative execution model (Steps 0–6) is defined in UCC-EXECUTION.txt §4.
This section summarizes implementation obligations only.

Step 0 — evaluate preconditions; if any fail → outcome=failed, failure_class=permanent, STOP
Step 1 — call observe(); if observation=failed or indeterminate → STOP, emit result
Step 2 — evaluate declaration against observed state; pure function, no side effects
Step 3 — call diff(); pure function
Step 4 — determine outcome from diff result and any extension block conditions
Step 5 — apply transition only if outcome=changed; call apply()
Step 6 — emit exactly one result message per UCC-EXECUTION.txt §3

For full semantics, field presence rules, and normative rules R1–R12,
see UCC-EXECUTION.txt §4.

------------------------------------------------------------

6. RESULT GENERATION

A ResultMessage MUST include:

- meta (with version = "2.0")
- observe.observed_before = S0 (when observation = ok)
- observe.diff = D (when observation = ok)
- result.observation
- result.outcome (when observation = ok)
- result.completion (when outcome = changed)
- result.failure_class (when outcome = failed)
- result.inhibitor (when outcome = unchanged)
- optional proof
- optional message (required when outcome = failed or observation ≠ ok)

The reported result MUST match reality.

------------------------------------------------------------

7. ERROR HANDLING

Errors MUST NOT:

- throw unreported exceptions
- partially modify state
- be hidden

Errors MUST:

- return observation=ok, outcome=failed with failure_class=retryable or permanent
- or return observation=failed (infrastructure failure)
- or return observation=indeterminate (state unclear)
- leave state unchanged
- be observable in the result

------------------------------------------------------------

8. DRY-RUN SUPPORT

Dry-run is mode = dry_run.

A dry-run is execution where:
- observe is performed
- apply is evaluated
- diff is computed
- no transition is applied (Step 5 MUST NOT execute)

Valid dry-run results:
- If D = ∅: outcome=converged
- If D ≠ ∅: outcome=unchanged, inhibitor=dry_run

Step 0 and Step 1 behavior is unchanged in dry-run:
- If precondition fails: outcome=failed, failure_class=permanent
- If O(W) = ⊗: observation=failed
- If O(W) = ?: observation=indeterminate

------------------------------------------------------------

9. ASYNC SUPPORT (mode=verify)

When mode = verify (Phase 2 verification):
- Step 2 is skipped; S1 is recovered from the verification_token
- Step 5 MUST NOT execute
- Valid outcomes: outcome=converged, outcome=changed completion=pending
  (in-flight re-issue), outcome=failed, observation=indeterminate, observation=failed

The verification token links Phase 2 back to the Phase 1 desired state and
equality semantics. See UCC-EXT-ASYNC.txt for the complete lifecycle.

------------------------------------------------------------

10. WHAT IMPLEMENTERS OFTEN GET WRONG

Common mistakes:

- performing state changes inside apply()
- skipping observation
- computing diff after mutation
- returning custom result values (observation, outcome, etc. outside closed enums)
- hiding partial failures
- coupling implementation to transport
- omitting completion when outcome = changed
- omitting failure_class when outcome = failed
- omitting inhibitor when outcome = unchanged

Any of these violates UCC.

------------------------------------------------------------

11. TRANSPORT AND INTEGRATION

UCC does not define transport.

Valid transports include:
- HTTP
- CLI
- message queues
- files
- in-memory calls

Transport MUST NOT affect semantics.

------------------------------------------------------------

12. SCALING AND COMPOSITION

UCC components compose by chaining:

Result(S1) → observe(S1) → Declaration(S2)

No additional orchestration logic is required.

------------------------------------------------------------

13. MINIMAL COMPLIANT IMPLEMENTATION CHECKLIST

An implementation is minimally compliant if:

- it implements observe(), apply(declaration, state), diff()
- it emits valid UCC/2.0 messages (three-field result model)
- it passes all eight mandatory test phases (see UCC-TESTS.txt for the complete list of mandatory phases)
- it produces deterministic results

------------------------------------------------------------

14. CONCLUSION

UCC implementations should be:

- small
- boring
- explicit
- deterministic

Complexity belongs outside the contract.
