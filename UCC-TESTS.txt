UCC-TESTS – Test Contract Specification
Universal Convergence Contract (UCC)

This document is NORMATIVE.
It defines the mandatory test structure that any UCC-compliant component MUST satisfy.

------------------------------------------------------------

CLASSIFICATION
--------------
Normative – Test Contract

------------------------------------------------------------

1. PURPOSE OF THIS DOCUMENT

UCC guarantees convergence correctness, determinism, and idempotence.
These properties are not verifiable by inspection alone.
They MUST be verified by tests.

This document defines:

- which test phases are mandatory for UCC compliance
- what each test phase must verify
- how test cases map to the UCC result model
- the minimal test runner requirements

NOTE: All test result assertions in this document MUST produce result messages
that validate against UCC-SCHEMA.txt. Schema validation is a prerequisite
for compliance certification.

------------------------------------------------------------

2. CORE PRINCIPLE

A UCC component is not compliant unless its full result space
is exercised by tests that fail when the behavior is broken.

Documentation, code review, and manual verification are insufficient.

------------------------------------------------------------

3. MANDATORY TEST PHASES

Every UCC-compliant component MUST implement the following test phases:

Phase 1 — HappyClean
Phase 2 — HappyIdempotent
Phase 3 — FailRetryable
Phase 4 — FailPermanent
Phase 5 — Unchanged
Phase 6 — Partial (multi-target components only; not-applicable with justification otherwise)
Phase 7 — FailObservation
Phase 8 — ObservationIndeterminate

Each phase maps to one or more results in the UCC result model.
Each phase MUST be independently executable.

------------------------------------------------------------

4. PHASE DEFINITIONS

------------------------------------------------------------

4.1 HappyClean

Purpose:
Verify that the component correctly transitions a target from
an absent or non-compliant state to the desired state.

Precondition:
Target does NOT satisfy the desired state before execution.

Expected result:
observation=ok, outcome=changed, completion=complete (or partial=true if
multi-target with at least one sub-target succeeding)

Required assertions:
- observation == ok
- outcome == changed
- completion == complete
- observed_before reflects the non-compliant state
- observed_after reflects the desired state
- diff is non-empty before execution
- diff is empty when re-observed after execution

------------------------------------------------------------

4.2 HappyIdempotent

Purpose:
Verify that applying the same declaration to an already-converged target
produces no change.

Precondition:
Target ALREADY satisfies the desired state.
Typically: run HappyClean first, then run HappyIdempotent on the same target.

Expected result:
observation=ok, outcome=converged

Required assertions:
- observation == ok
- outcome == converged
- observed_before satisfies the desired state
- diff is empty
- no mutation occurred (observed state is identical to input state)

------------------------------------------------------------

4.3 FailRetryable

Purpose:
Verify that the component correctly reports a transient failure
and classifies it as retryable.

Precondition:
A transient condition is simulated or injected:
- network unreachable
- locked resource
- timeout condition

Expected result:
observation=ok, outcome=failed, failure_class=retryable

Required assertions:
- observation == ok
- outcome == failed
- failure_class == retryable
- message is non-empty and describes the condition
- observed_before is present (if observation succeeded before the failure)
- observed_after is absent
- no partial state change occurred

Note:
If the component cannot produce a transient failure condition,
this phase MUST be documented as not-applicable with justification.
It may NOT be silently omitted.

------------------------------------------------------------

4.4 FailPermanent

Purpose:
Verify that the component correctly identifies a structurally
impossible desired state and classifies it as permanent.

Precondition:
A permanent impossibility is established:
- target type incompatible
- invalid or self-contradictory desired_state
- required capability not present in environment

Expected result:
observation=ok, outcome=failed, failure_class=permanent

Required assertions:
- observation == ok
- outcome == failed
- failure_class == permanent
- message is non-empty and describes the permanent cause
- no retry is triggered by the component

Note:
Every UCC component MUST define at least one permanent failure case.
If no permanent failure is possible by design, this MUST be documented explicitly.

------------------------------------------------------------

4.5 Unchanged

Purpose:
Verify that when a transition is inhibited (dry_run, whatif, or policy),
the component returns outcome=unchanged with a non-empty diff and the
correct inhibitor field.

Precondition:
Target does NOT satisfy desired state. A policy or dry-run mode is active.

Expected result:
observation=ok, outcome=unchanged

Required assertions:
- observation == ok
- outcome == unchanged
- inhibitor is one of: dry_run, whatif, policy
- diff is non-empty
- no state mutation occurred

------------------------------------------------------------

4.6 Partial

Purpose:
Verify that when some sub-targets succeed and others fail, the component
returns outcome=changed, partial=true with accurate counts.
NOTE: For single-target components, document this phase as not-applicable
with justification. It may NOT be silently omitted.

Precondition:
Multiple sub-targets exist; conditions are arranged so some succeed and
some fail.

Expected result:
observation=ok, outcome=changed, completion=complete, partial=true

Required assertions:
- observation == ok
- outcome == changed
- partial == true
- partial_detail is present
- partial_detail.total = partial_detail.converged + partial_detail.changed + partial_detail.failed
- observed_after is present

------------------------------------------------------------

4.7 FailObservation

Purpose:
Verify that when the observation infrastructure itself fails (⊗), the
component returns observation=failed (not observation=indeterminate).

Precondition:
Observation mechanism is made unreachable or broken (not the target itself).

Expected result:
observation=failed

Required assertions:
- observation == failed
- outcome is absent
- observed_before is absent
- diff is absent
- observed_after is absent
- message is non-empty

------------------------------------------------------------

4.8 ObservationIndeterminate

Purpose:
Verify that when observation runs but produces an indeterminate state (?),
the component returns observation=indeterminate.

Precondition:
Observation returns an ambiguous or indeterminate state.

Expected result:
observation=indeterminate

Required assertions:
- observation == indeterminate
- outcome is absent
- diff is absent
- observed_after is absent
- message is non-empty

------------------------------------------------------------

5. EXTENSION TEST PHASES (MANDATORY WHEN BLOCK IS DECLARED)

When a declaration block from a UCC/2.0 extension is used,
the corresponding test phases below become MANDATORY.
They are not required for implementations that do not declare
the corresponding extension block.

------------------------------------------------------------

5.1 Equality Extension (UCC-EXT-EQUALITY.txt)

EQ-1 — VolatileFieldExclusion
  Verify that a field declared in `volatile_fields` does not
  appear in the diff and does not trigger a transition.
  Required assertions:
  - diff does not contain the volatile field
  - outcome is converged when only volatile fields differ

EQ-2 — SemanticModeProof
  Verify that when mode = semantic, the proof contains a
  `semantic_basis` entry with all required fields including
  a reverse-DNS scoped rule_id.
  Required assertions:
  - proof.semantic_basis.rule_id contains at least one dot
  - proof.semantic_basis.rule_version is non-empty
  - If proof.semantic_basis.fields_considered_equivalent is present, it MUST be non-empty.

------------------------------------------------------------

5.2 Concurrency Extension (UCC-EXT-CONCURRENCY.txt)

CC-1 — OptimisticConflict
  Verify that a version conflict is detected and the declared
  `conflict` strategy is applied correctly.
  Required assertions:
  - conflict = fail → outcome=failed, failure_class=retryable
  - conflict = skip → outcome=unchanged, inhibitor=policy
  - conflict = retry (exhausted) → outcome=failed, failure_class=retryable
  - proof.concurrency.conflict_detected = true

CC-2 — PessimisticLockTimeout
  Verify that when the lock cannot be acquired within lock_timeout_ms,
  the result is outcome=failed, failure_class=retryable.
  Required assertions:
  - outcome == failed
  - failure_class == retryable
  - message is non-empty
  - no state mutation occurred

CC-3 — PessimisticAsyncExclusion
  Verify that declaring strategy = pessimistic on an inherently
  async target produces outcome=failed, failure_class=permanent.
  Required assertions:
  - outcome == failed
  - failure_class == permanent
  - message references async incompatibility

CC-4 — ConcurrencyProof
  Verify that proof.concurrency is present and non-empty for
  optimistic and pessimistic strategies.
  Required assertions:
  - proof.concurrency.strategy matches declared strategy
  - proof.concurrency.version_before is present (optimistic)

------------------------------------------------------------

5.3 Async Extension (UCC-EXT-ASYNC.txt)

AS-1 — PendingInitiation
  Verify that a long-running operation produces outcome=changed,
  completion=pending with a valid verification block.
  Required assertions:
  - outcome == changed
  - completion == pending
  - verification.token is non-empty
  - verification.expires_at is a valid future timestamp
  - observed_after is absent

AS-2 — VerificationConverged
  Verify that a completed operation produces outcome=converged
  on the verification cycle (mode=verify).
  Required assertions:
  - Phase 2 outcome is converged
  - observed_after is present when outcome=changed, completion=complete

AS-3 — VerificationExpiry
  Verify that an expired token produces outcome=failed, failure_class=permanent
  without executing observation.
  Required assertions:
  - outcome == failed
  - failure_class == permanent
  - message references token expiry

AS-4 — InFlightReissue
  Verify that an in-progress operation on Phase 2 produces
  outcome=changed, completion=pending with a re-issued token.
  Required assertions:
  - outcome == changed
  - completion == pending
  - expires_at is not extended beyond original expiry

AS-5 — ExpiryBound (if policy.max_lifetime_ms declared)
  Verify that expires_at does not exceed Phase 1 timestamp
  plus max_lifetime_ms.
  Required assertions:
  - expires_at ≤ phase1_timestamp + max_lifetime_ms

AS-6 — DefaultExpiryBound (when max_lifetime_ms absent)
  Verify that when policy.max_lifetime_ms is absent, expires_at does not
  exceed Phase 1 timestamp plus 86400000 ms (24 hours).
  Required assertions:
  - expires_at ≤ phase1_timestamp + 86400000 ms

------------------------------------------------------------

5.4 Compensation Extension (UCC-EXT-COMPENSATION.txt)

CP-1 — AutoCompensationSuccess
  Verify that when trigger = on_failure or on_partial and the
  trigger condition is met, compensation runs and the result
  includes the primary outcome plus a compensation block with
  status = converged or changed.
  Required assertions:
  - primary outcome (observation, outcome, failure_class or partial) is preserved
  - compensation.status is present and is converged or changed
  - compensation.observed_after_compensation is present

CP-2 — AutoCompensationFailure
  Verify that when compensation itself fails, the original primary
  outcome is retained and the compensation block carries status=failed.
  Required assertions:
  - primary outcome is preserved (not replaced)
  - compensation.status == failed
  - compensation.failure_class is present
  - compensation.message is non-empty

CP-3 — ManualCompensationSignal
  Verify that when trigger = manual and the primary fails or is
  partial, compensation_available is present with expires_at.
  Required assertions:
  - compensation_available.declaration_id is non-empty
  - compensation_available.expires_at is a valid future timestamp

CP-4 — CompensationExpiryEnforcement
  Verify that executing compensation after compensation_available.
  expires_at is rejected.
  Required assertions:
  - outcome == failed
  - failure_class == permanent
  - message references compensation expiry

CP-5 — CompensationContextRejectionMessage
  When trigger = manual, compensation_context is present, and the
  compensation is rejected (expired or invalid), verify the message
  references original_interaction_id.
  Required assertions:
  - outcome == failed
  - failure_class == permanent
  - message contains the original_interaction_id value

------------------------------------------------------------

6. INTERACTION TEST PHASES (MANDATORY WHEN MULTIPLE EXTENSIONS ARE COMBINED)

When two or more extension blocks are declared together, the applicable
interaction test phases below become MANDATORY.
See UCC-COMPOSITION.txt for the normative rules under test.

------------------------------------------------------------

6.1 Pessimistic + Async (COMP-1)

INT-1 — PessimisticAsyncRejection
  Declare strategy = pessimistic on a target that requires async execution.
  Verify the result is outcome=failed, failure_class=permanent before any observation.
  Required assertions:
  - outcome == failed
  - failure_class == permanent
  - message references async incompatibility with pessimistic strategy
  - no observation was attempted

------------------------------------------------------------

6.2 Nested Compensation (COMP-2)

INT-2 — NestedCompensationRejection
  Declare a compensation declaration whose declaration itself
  contains a `compensates` field.
  Verify the result is outcome=failed, failure_class=permanent before any observation.
  Required assertions:
  - outcome == failed
  - failure_class == permanent
  - message references nested compensation prohibition

------------------------------------------------------------

6.3 Async Compensation (COMP-3)

INT-3 — AsyncCompensationRejection
  Simulate a compensation cycle that would return completion=pending.
  If a compensation cycle returns completion=pending, the overall result retains the
  original primary outcome and compensation.status = failed with a message
  referencing async compensation prohibition.
  Required assertions:
  - primary outcome is unchanged (not replaced)
  - compensation.status == failed
  - compensation.message references async compensation prohibition

------------------------------------------------------------

6.4 Async + Compensation (COMP-4, COMP-5)

INT-4 — DeferredCompensationTrigger
  Phase 1 produces completion=pending. Verify compensation does NOT fire.
  Resolve the verification cycle to a trigger condition (partial=true or failure).
  Verify compensation fires at the resolving verification cycle.
  Required assertions:
  - Phase 1: outcome=changed, completion=pending; no compensation block
  - Resolving Phase 2: primary outcome preserved; compensation block present

------------------------------------------------------------

6.5 Async + Equality (COMP-9)

INT-5 — EqualityPropagationToVerification
  Declare a volatile_field in the equality block at Phase 1.
  In Phase 2 verification (mode=verify), expose a changed value in that volatile field.
  Verify the volatile field does not appear in the verification diff
  and does not prevent outcome=converged from being returned.
  Required assertions:
  - volatile field absent from Phase 2 diff
  - outcome is converged despite volatile field change
  Normative basis: UCC-COMPOSITION.txt COMP-9 (equality semantics propagate to verification).

------------------------------------------------------------

6.6 Equality + Optimistic Concurrency (COMP-10, COMP-10-C)

INT-6 — VersionTokenVolatileRequirement
  Declare strategy = optimistic without including the version token field
  in equality.volatile_fields.
  Verify the result is outcome=failed, failure_class=permanent.
  Required assertions:
  - outcome == failed
  - failure_class == permanent
  - message identifies the undeclared version token field

------------------------------------------------------------

7. OPTIONAL TEST PHASES

The following phases are recommended but not mandatory for compliance.
They extend the mandatory phases with additional sub-scenarios.

7.1 Additional FailRetryable sub-scenarios
Extend Phase 3 with distinct transient conditions beyond the baseline case:
- network unreachable (distinct from lock timeout)
- resource locked by a concurrent writer
- upstream dependency timeout

7.2 Additional FailPermanent sub-scenarios
Extend Phase 4 with distinct impossibility types beyond the baseline case:
- invalid or self-contradictory desired_state
- required capability absent from environment
- target type structurally incompatible

7.3 Multi-target Partial sub-scenarios with count verification
Extend Phase 6 (Partial) with additional sub-target count combinations:
- exactly one sub-target fails (all others succeed)
- majority of sub-targets fail (minority succeed)
- Verify partial_detail.total = converged + changed + failed in each case

7.4 Inhibitor sub-scenarios
Extend Phase 5 (Unchanged) with distinct inhibitor types not covered by
the mandatory baseline:
- dry_run mode active → inhibitor = dry_run
- whatif mode active → inhibitor = whatif
- policy-based inhibition → inhibitor = policy
Verify that each inhibitor value is correctly set in the inhibitor field.

7.5 Dry-run mode sub-scenarios
Verify mode=dry_run behavior:
- If D=∅ in dry_run: outcome=converged
- If D≠∅ in dry_run: outcome=unchanged, inhibitor=dry_run
- No state mutation in either case

------------------------------------------------------------

8. TEST NAMING CONVENTION

Test cases SHOULD follow this naming pattern:

  [Component]_[Phase]_[Scenario]

Examples:
  FileRemove_HappyClean_FilePresent
  FileRemove_HappyIdempotent_FileAlreadyAbsent
  FileRemove_FailRetryable_FileLocked
  FileRemove_FailPermanent_InvalidPath
  PackageInstall_FailPermanent_UnsupportedOS

------------------------------------------------------------

9. TEST RUNNER REQUIREMENTS

A UCC test runner MUST:

1. Execute all mandatory phases
2. Execute phases in order: HappyClean → HappyIdempotent → FailRetryable → FailPermanent → Unchanged → Partial → FailObservation → ObservationIndeterminate
3. Restore the environment to its precondition state between phases
4. Fail if any mandatory phase is absent (or undocumented as not-applicable)
5. Fail if any assertion fails
6. Report which phase and assertion failed

A UCC test runner MUST NOT:
- skip any mandatory phase silently
- treat a missing phase as passing
- allow phase execution order to be arbitrary

------------------------------------------------------------

10. PHASE ORDERING RATIONALE

HappyClean MUST run before HappyIdempotent.
Reason: HappyIdempotent requires a compliant target to exist.
HappyClean establishes that target.

FailRetryable and FailPermanent are independent of HappyClean
but MUST execute in a clean environment to avoid state contamination.

------------------------------------------------------------

11. RELATION TO TEST-ENFORCED-DEVELOPMENT.txt

This document extends TEST-ENFORCED-DEVELOPMENT.txt with UCC-specific
test structure requirements.

TEST-ENFORCED-DEVELOPMENT.txt defines: every behavioral unit MUST have a test.
UCC-TESTS.txt defines: every UCC contract MUST cover all mandatory phases.

These requirements are complementary and non-overlapping.

------------------------------------------------------------

12. COMPLIANCE CRITERION

A UCC component is test-compliant if and only if:

- All mandatory phases are implemented (or documented as not-applicable with justification)
- Each phase includes at least one test case
- All required assertions are present and valid
- All assertions fail when the behavior is broken
- The test runner passes all phases without exception

------------------------------------------------------------

13. NORMATIVE SUMMARY

Eight test phases are mandatory for every UCC component:

  HappyClean              — verifies correct convergence from non-compliant state
  HappyIdempotent         — verifies safe re-execution on already-compliant state
  FailRetryable           — verifies correct transient failure classification (failure_class=retryable)
  FailPermanent           — verifies correct permanent failure classification (failure_class=permanent)
  Unchanged               — verifies inhibited transition with non-empty diff and inhibitor field
  Partial                 — verifies mixed sub-target outcome (n/a with justification if single-target)
  FailObservation         — verifies observation=failed when observation infrastructure fails
  ObservationIndeterminate — verifies observation=indeterminate when observation produces indeterminate state

A component without these phases cannot claim UCC compliance.
