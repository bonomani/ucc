1. PURPOSE OF THIS DOCUMENT

This document provides:

- a formal mathematical model of UCC
- a proof of minimality
- a proof of universality
- a proof of determinism

It exists to show that UCC is not a convention,
but a formally grounded model.

------------------------------------------------------------

2. UNIVERSE OF DISCOURSE

Let:

- W be the real world (systems, resources, environments)
- S be the set of all observable states
- R be the finite set of possible results

S is defined as the set of all finite, serializable,
and comparable structures observable from W.

------------------------------------------------------------

3. OBSERVATION FUNCTION

Define the observation function:

O : W → S ∪ {?} ∪ {⊗}

Where ? denotes an indeterminate state (observation ran, result unclear).
Where ⊗ denotes observation infrastructure failure (observation did not run).

Properties of O:

- O is deterministic
- O produces a complete observable state, or ?, or ⊗
- O has no side effects

Observation is the only allowed entry point
from the real world into the contract.

In the result model:
  O(W) ∈ S  → observation = ok
  O(W) = ?  → observation = indeterminate
  O(W) = ⊗  → observation = failed

------------------------------------------------------------

4. DECLARATION FUNCTION

Define a declaration as a partial function:

D : S → S ∪ {⊥}

Where:
- D(s) = desired state derived from observed state s
- ⊥ denotes a transition that is structurally impossible

Declaration is declarative.
It does not encode actions or procedures.

------------------------------------------------------------

5. STATE DIFFERENCE (DIFF)

Define a difference function:

Δ : S × S → D

With the invariant:

Δ(s0, s1) = ∅  ⇔  s0 ≡ s1

Where:
- D is the set of all possible state differences
- ∅ denotes an empty difference

Diff is purely comparative.
It has no operational meaning.

------------------------------------------------------------

6. RESULT MODEL
------------------------------------------------------------

The result is expressed through the three-field model defined in
UCC-EXECUTION.txt §2. The following formalizes it mathematically.

Let:
  Obs  = { ok, indeterminate, failed }
  Out  = { converged, changed, unchanged, failed }
  Comp = { complete, pending }
  FC   = { retryable, permanent }
  Inh  = { dry_run, whatif, policy }

A result R is a tuple drawn from the valid combinations:

  (ok, converged,  —,        —,         —      )
  (ok, changed,    complete, —,         —      )   partial modifier may apply
  (ok, changed,    pending,  —,         —      )
  (ok, unchanged,  —,        —,         <Inh>  )
  (ok, failed,     —,        <FC>,      —      )
  (indeterminate, —, —, —, —)
  (failed,        —, —, —, —)

All other combinations are forbidden.
For field presence rules and consumer obligations, see UCC-EXECUTION.txt §2
and UCC-STATUS.txt.

------------------------------------------------------------

7. TRANSITION SEMANTICS

Given an observed state s0 ∈ S (or ?, or ⊗ if observation fails):

1. Observe: s0 = O(W)

Cases (first match wins):

-- Precondition evaluation is a Step 0 hard gate (see UCC-EXECUTION.txt §4 Step 0).
-- If any precondition fails, the cycle terminates before Step 1.
- If precondition not satisfied                           → observation=ok, outcome=failed, failure_class=permanent
  (preconditions are evaluated before observation — see UCC-EXECUTION.txt Step 0)
- If O(W) = ⊗                                            → observation = failed
- If O(W) = ?                                            → observation = indeterminate
- If D(s0) = ⊥ and failure is permanent                 → observation=ok, outcome=failed, failure_class=permanent
- If D(s0) = ⊥ and failure is transient                 → observation=ok, outcome=failed, failure_class=retryable
- If Δ(s0, D(s0)) = ∅                                   → observation=ok, outcome=converged
- If Δ(s0, D(s0)) ≠ ∅ and transition not applied       → observation=ok, outcome=unchanged, inhibitor=<reason>
- If Δ(s0, D(s0)) ≠ ∅, applied, not yet verifiable     → observation=ok, outcome=changed, completion=pending
- If Δ(s0, D(s0)) ≠ ∅, applied, partial failure        → observation=ok, outcome=changed, completion=complete, partial=true
- If Δ(s0, D(s0)) ≠ ∅, applied, all succeeded          → observation=ok, outcome=changed, completion=complete

-- Ordering matches UCC-EXECUTION.txt Step 4 decision table.
-- Compensation is reported alongside the preserved primary outcome (see UCC-EXT-COMPENSATION.txt).

------------------------------------------------------------

8. IDEMPOTENCE PROPERTY

For all s ∈ S:

apply(D, apply(D, s)) = apply(D, s)

This guarantees:

- safe retries
- replayability
- absence of cumulative side effects

------------------------------------------------------------

9. DETERMINISM THEOREM

For any two executions E1 and E2:

If:
- O1 = O2
- D1 = D2

Then:
- R1 = R2

Formally:

(O(W1) = O(W2)) ∧ (D1 = D2) ⇒ (R1 = R2)

------------------------------------------------------------

10. UNIVERSALITY PROPOSITION

For any software transaction T,
there exists a tuple (O, D, Δ) such that:

T ≡ O ∘ D ∘ Δ → R

This holds for:

- function calls
- API invocations
- database transactions
- scripts
- events
- workflows
- agents
- AI systems

------------------------------------------------------------

11. MINIMALITY PROOF (SKETCH)

UCC uses exactly four primitives:

- observation
- declaration
- diff
- result

Removing any primitive breaks universality:

- no observation → no baseline
- no declaration → no objective
- no diff → no convergence
- no result → no determinism

Therefore, the model is minimal.

------------------------------------------------------------

12. CONCLUSION

UCC is mathematically:

- deterministic
- idempotent
- universal
- minimal

This formal model proves that UCC/2.0
is not dependent on language, protocol, or runtime,
but on fundamental properties of computation itself.
