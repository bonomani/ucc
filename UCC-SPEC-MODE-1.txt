UCC-SPEC-MODE-1: Execution Modes and Wrapper Pattern
Clear and Minimal Specification of UCC Modes and Wrapper Pattern

This document is a SIMPLIFIED and CLARIFIED specification.
Its goal is precision, not exhaustiveness.
Each concept is stated once, plainly, with no ambiguity.

------------------------------------------------------------

CLASSIFICATION
--------------
Product/Planning – Execution Mode Specification

------------------------------------------------------------

============================================================
1. PROBLEM STATEMENT
============================================================

This specification addresses the need to:

- introduce UCC (formal declaration and result)
- maintain deterministic and testable behavior

These goals require:

- explicit execution mode selection
- strict separation of external contracts
- prohibition of implicit or heuristic behavior

============================================================
2. UCC MODE SELECTOR (CANONICAL)
============================================================

There is exactly ONE valid mode selector:

  ucc = full

No other values or selectors are permitted.

The selected mode uniquely determines:
- how input is interpreted
- how output is produced

============================================================
3. SEMANTICS OF THE MODE
============================================================

------------------------------------------------------------
3.1 ucc=full
------------------------------------------------------------

Definition:
- full UCC execution mode

Input contract:
- UCC DeclarationMessage (JSON via stdin)

Output contract:
- UCC ResultMessage (JSON on stdout)

stderr:
- runtime or transport errors only

Exit codes:
- MUST NOT carry business semantics

Operational meaning:
- complete formal contract
- deterministic and auditable execution

============================================================
4. ABSOLUTE RULES (NORMATIVE)
============================================================

MODE-1 (MUST):
- Each invocation MUST execute in exactly one UCC mode.

MODE-2 (MUST):
- The selected mode MUST fully determine BOTH input and output contracts.

MODE-3 (MUST NOT):
- Mixed contracts MUST NOT occur.

MODE-4 (MUST NOT):
- Implicit, inferred, or auto-detected modes MUST NOT exist.

MODE-5 (MUST):
- Same invocation with the same input MUST always produce the same observable behavior.

============================================================
5. FORBIDDEN MODES
============================================================

The following modes MUST NOT exist:

- ucc=output
- ucc=both
- any implicit or heuristic mode

Rationale:
- UCC output without UCC input is invalid
- "both" implies mixed contracts
- ambiguity breaks determinism, safety, and testability

============================================================
6. WRAPPER (ADAPTER) PATTERN
============================================================

------------------------------------------------------------
6.1 Rationale
------------------------------------------------------------

In many environments:

- system binaries (e.g. /bin/cp) cannot be replaced
- underlying tools must remain unchanged

In these cases:
- a wrapper (adapter) MUST be used

------------------------------------------------------------
6.2 Wrapper Behavior
------------------------------------------------------------

Example wrapper:
  cp-ucc

Behavior:

- ucc=full
  → read DeclarationMessage
  → invoke cp internally
  → emit ResultMessage only

------------------------------------------------------------
6.3 Wrapper Guarantees (MUST)
------------------------------------------------------------

A conforming wrapper MUST guarantee:

- stdout is NEVER mixed between non-UCC and UCC formats
- ResultMessage appears ONLY in ucc=full
- identical declaration produces identical results
- removing the wrapper fully restores the original system

============================================================
7. DESIGN RATIONALE
============================================================

This design works because it cleanly separates:

- declaration: what must be true
- execution: how the state is achieved
- contract: how results are observed

============================================================
8. SUMMARY
============================================================

A single, explicit mode selector (ucc=full) removes all ambiguity.
Input and output contracts are fully determined by the mode.
Determinism and auditability are guaranteed by construction.
