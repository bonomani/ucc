PURPOSE
-------
Normative semantic requirements for any orchestrator implementing UCC/GCC/GIC
contract execution. These rules are tool-agnostic and apply regardless of
implementation language, platform, or deployment model.

------------------------------------------------------------
1. FOUNDATIONAL PRINCIPLES
------------------------------------------------------------

Any compliant orchestrator MUST enforce the following:

1. All boundary interactions MUST be BISS-classified (GIC, GCC, or UCC).
2. All internal logic MUST follow RIG.
3. No implicit retry is ever permitted.
4. No heuristic decisions are permitted.
5. Orchestration logic MUST be generic and fixed — it MUST NOT contain
   business logic.

These principles are not optional. Deviation from any of them means the
orchestrator is not BISS-RIG compliant.

------------------------------------------------------------
2. CONTRACT TYPE EXECUTION SEMANTICS
------------------------------------------------------------

The orchestrator MUST process each step according to its declared contract
type. The following rules are authoritative.

GIC (Generic Interaction Core)
----------------------------------
- The orchestrator MUST execute the step.
- Exit code MUST be ignored.
- Failure MUST be logged and then ignored — it MUST NOT affect workflow
  continuation.
- A GIC step MUST NOT be replayed under any circumstance.
- A GIC step MUST NOT block a subsequent step.

GCC (Generic Contract Core)
-------------------------------
- The orchestrator MUST execute the step.
- The exit code is authoritative and MUST be the sole basis for
  success/failure determination.
- If a GCC step fails, the orchestrator MUST stop workflow execution
  immediately.
- A successful GCC step MUST NOT be replayed.
- No retry of a failed GCC step is implied or permitted without explicit
  re-invocation.

UCC (Universal Convergence Contract)
--------------------------------------
- The orchestrator MUST execute the step.
- Convergence is determined by exit code combined with a state check;
  neither alone is sufficient.
- If a UCC step has not converged, it MUST be marked for replay.
- UCC steps are the only steps eligible for replay.
- Replay MUST continue until the step reaches a terminal converged state.

------------------------------------------------------------
3. REPLAY MODEL SEMANTICS
------------------------------------------------------------

Replay is ALWAYS explicit. No implicit background retry is permitted.

When replay/reconcile is invoked, the orchestrator MUST:
- Reload persisted state.
- Re-execute only UCC steps that have not yet converged.
- NEVER replay a GIC step.
- NEVER replay a GCC step that has already succeeded.

A replay run is semantically identical to the original run with respect to
contract enforcement — all rules in Section 2 apply equally during replay.

------------------------------------------------------------
4. RESULT AND STATUS HANDLING RULES
------------------------------------------------------------

Exit code interpretation MUST follow contract type exclusively:

  GIC  : exit code is ignored; step is always treated as complete.
  GCC  : exit code is authoritative; non-zero means workflow stops.
  UCC  : exit code combined with state check determines convergence.

The orchestrator MUST NOT parse business output (stdout/stderr content)
to determine step outcome. Only the exit code and, for UCC, the
accompanying state check are permitted inputs to the result determination.

------------------------------------------------------------
5. INTERNAL ORCHESTRATOR REQUIREMENTS (RIG)
------------------------------------------------------------

The orchestrator MUST:
- Log every state transition.
- Assert valid workflow structure before execution begins.
- Detect and reject invalid dependency graphs.
- Crash fast on internal inconsistency — internal bugs MUST NOT be
  silently absorbed or treated as workflow failures.
- Be fully testable in isolation.

Internal bugs are categorically not workflow failures and MUST NOT be
reported as such.

------------------------------------------------------------
6. WHAT THESE RULES GOVERN
------------------------------------------------------------

These rules govern the SEMANTIC CONTRACT between the orchestrator and the
steps it executes. They apply to:

- Any language runtime implementing the orchestrator.
- Any platform (CLI, server, embedded, AI agent loop).
- Any workflow definition format (YAML, JSON, programmatic).

They do NOT govern:
- How the CLI surface is designed.
- How workflow files are formatted.
- How state is persisted internally.
- How observability output is structured.

Those concerns are addressed in the relevant product specification
(e.g., BFO-CLI-MVP.txt).

END OF DOCUMENT
